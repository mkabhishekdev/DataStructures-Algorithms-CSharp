5. Difference between git merge and git rebase?

🔹 git merge

What it does: Takes another branch (say main) and merges it into your branch (feat/data-structures).

It creates a new commit called a "merge commit".

Keeps the full history (with all branches).

Example:
git checkout feat/data-structures
git merge main


Result in history:

A---B---C (main)
         \
          D---E---M (feat/data-structures)


M is the merge commit.

Easy, safe, preserves context.

But history looks more “branchy” and can get messy in big teams.


🔹 git rebase

What it does: Takes your commits and “replays” them on top of another branch (say main).

Rewrites commit history to make it look like you built your feature after the latest main.

Example:
git checkout feat/data-structures
git rebase main


Result in history:

A---B---C (main)
             \
              D'---E' (feat/data-structures rebased)


No merge commit.

Cleaner, linear history.

But: rewrites commits (D → D’, E → E’), which can cause issues if others already pulled your branch.

✅ When to Use What

Merge:

Collaborative work
Safe, no commit rewriting
Good for resolving conflicts together

Rebase:

Personal branches not yet shared with team
You want a clean, linear history
Often used before merging a PR (git rebase main then squash commits)

8. Git Repo Hygiene SOP (bin/obj/.gitignore Cleanup Guide)
 
PART 1 — Understand the Core Rule
--------------------------------------------
🔹 Important Truth

.gitignore prevents NEW junk files from being tracked.
It does NOT remove files that were already committed earlier.

So if a repo was created before .gitignore existed, junk files may already be tracked.

PART 2 — First Audit (Never Guess)
------------------------------------------

Before touching anything, check what Git is currently tracking.

Step 1 — Check if .gitignore is tracked
git ls-files | grep .gitignore


If it prints .gitignore → good.
If nothing → commit it first.

Step 2 — Check if junk is already tracked
git ls-files | grep '/bin/'
git ls-files | grep '/obj/'
git ls-files | grep -E '\.(dll|pdb|exe|cache)$'


If nothing prints → repo is clean ✅
If files print → hygiene needed ❌


PART 3 — Safe Cleanup Strategy (Professional Way)
----------------------------------------------------
✅ Always create a cleanup branch

Never clean directly on main.

git checkout -b chore/repo-hygiene-cleanup

✅ Ensure .gitignore has proper rules

For .NET projects, minimum:

bin/
obj/
.vs/
*.user
*.suo


Commit if updated:

git add .gitignore
git commit -m "chore: ensure proper gitignore rules"

✅ Perform Full Index Rebuild (Most Reliable Method)

This is the safest complete cleanup method.

git rm -r --cached .
git add .


What this does:

Removes everything from Git tracking (not disk)

Re-adds only files allowed by .gitignore

Think of it as:

“Rebuild Git tracking from scratch using .gitignore rules.”

✅ Verify Before Commit
git ls-files | grep '/bin/'
git ls-files | grep '/obj/'
git ls-files | grep -E '\.(dll|pdb|exe|cache)$'


Nothing should print.

If clean → proceed.

✅ Commit Cleanup
git commit -m "cleanup: enforce repository hygiene via gitignore"

✅ Push Cleanup Branch
git push origin chore/repo-hygiene-cleanup

✅ Merge Into Main

Option A (Recommended):
Create PR → Review → Merge

Option B (CLI):

git checkout main
git pull
git merge chore/repo-hygiene-cleanup
git push

✅ Delete Cleanup Branch (Optional)
git branch -d chore/repo-hygiene-cleanup
git push origin --delete chore/repo-hygiene-cleanup


PART 4 — Handling Modify/Delete Conflicts (If They Appear)
------------------------------------------------------------------------
If you see:

CONFLICT (modify/delete): file deleted in branch and modified in HEAD


It means:

main still has junk

your cleanup deleted junk

Git is asking what to keep

Solution:

Accept deletion:

git rm <file_or_folder>
git commit


Push again.


PART 5 — Advanced Repo Health (Optional Polishing)
---------------------------------------------------------------------
🔹 Shrink Repo Size

After large deletions:

git gc


This cleans and compresses Git objects.

🔹 Remove Large Files from Entire History (Advanced)

If huge binaries were committed long ago:

Use:

git filter-repo


(Only if necessary — this rewrites history.)

🔹 Prevent Future Junk (Pre-Commit Hook)

Create file:

.git/hooks/pre-commit


Add:

#!/bin/sh
if git diff --cached --name-only | grep -E '\.(dll|pdb|exe)$'; then
  echo "❌ Build artifacts cannot be committed."
  exit 1
fi


Make executable:

chmod +x .git/hooks/pre-commit


Now Git blocks accidental junk commits.

🧭 Quick Mental Model (Save This)
Problem	Root Cause	Fix
bin/obj in repo	committed before .gitignore	rebuild index
rebase conflicts on obj	tracked build artifacts	remove from tracking
modify/delete conflict	cleanup vs old main	accept deletion
large repo size	old binaries in history	git gc or filter-repo
🧑‍💼 Lead Engineer Principle

When cleaning repo:

Audit first

Never guess

Work in branch

Rebuild index

Verify before commit

Merge cleanly

🧠 Simple Analogy

Think of Git tracking like:

.gitignore = Security guard at the door

Old tracked files = People already inside the building

Adding a guard doesn’t remove people already inside.

Index rebuild = Evacuate building and let guard allow only correct people back in.

🧠 When is this used?

# 5. Create PR → merge into main
# After merge, production can deploy safely
🔑 Key Idea

Hotfix branches always start from:

Not from feature branches.


🍒 2️⃣ CHERRY-PICK PRODUCTION FIX (Apply Fix Across Branches)
------------------------------------------------------------------------
🧠 When is this used?
You fixed a bug in main, but the same fix is also needed in:
release branch
feature branch
older maintenance branch
Instead of rewriting code again → cherry-pick the exact commit.


🪜 Steps
-------------
Step 1: Find commit hash
git log --oneline

Example:
a1b2c3 fix: handle null payment response

Step 2: Apply fix to another branch
git checkout release/v1.2
git cherry-pick a1b2c3

Now the same fix exists in release branch 🎯
⚠️ If conflict occurs
# resolve conflicts manually
git add .
git cherry-pick --continue

🔑 Key Idea
Cherry-pick = copy one specific commit to another branch

Used heavily in:
production maintenance
multi-version enterprise apps


🧨 3️⃣ REVERTING A BAD RELEASE SAFELY
--------------------------------------------------
🧠 When is this used?
You deployed new release → production breaks → must undo without deleting history.

Never use git reset in shared repos ❌
Use git revert ✅

🪜 Steps 
-----------
Step 1: Identify bad commit
git log --oneline

Example:
d9f8e7 feat: new payment retry logic  <-- caused production issue

Step 2: Revert safely
git revert d9f8e7

This creates a new commit:
revert: undo payment retry logic

History remains intact ✔️
Production becomes stable ✔️

If reverting a merge commit
git revert -m 1 <merge_commit_hash>

(Advanced case — rarely needed initially)

🧠 Model (Very Important)
Situation	                                          Correct-Action
Urgent prod bug	                                  Hotfix branch from main
Same fix needed in multiple branches	             Cherry-pick commit
Release broke production	                        Revert commit safely

----------------------------
HOTFIX:

CHERRY-PICK:
find commit → checkout target branch → git cherry-pick <hash>
